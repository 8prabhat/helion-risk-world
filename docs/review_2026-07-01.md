# HelionRiskWorld — 360° Review (2026-07-01)

Scope: model architecture, data pipeline, training, evaluation/calibration/backtest, and prediction/inference.
Method: five independent deep-read passes (read-only, no fixes applied), each covering one area and hunting for
both implementation bugs and ML-methodology soundness issues (leakage, look-ahead bias, train/serve skew).
Findings below are merged and ranked by severity. File:line references are as of this review's snapshot.

---

## CRITICAL

### C1. Same-day EOD regime data leaks into every intraday bar
**File:** `src/helion_risk_world/data/daily_context_loader.py:41-48`, `scripts/fetch_free_data.py:184-231`, `src/helion_risk_world/data/regime_context_builder.py:135-142`

`DailyContextLoader.get()` resolves a query date via `searchsorted(..., side="right")`, which returns **today's**
row, not yesterday's — confirmed by `tests/test_data_pipeline.py:270-280`, which explicitly asserts same-day
lookup. Only `fii_dii_net` gets a 1-day lag shift in `build_daily_context`; `usdinr`, `crude` (Yahoo close), and
`atm_iv_pct`/`iv_skew_pct`/`pc_oi_ratio`/`basis` (NSE EOD bhavcopy) are stored under their own observation date
with **no lag**. A 09:20 IST decision on day D therefore trains on day-D's *closing* USD/INR, WTI close, and
EOD-derived IV/PCR/basis — none of which exist yet at 09:20 live. This is genuine look-ahead bias baked into
every training row, not an edge case, and would silently inflate backtest performance every single day.

**Fix direction:** lag all daily_context fields by one full trading day (or to true publication time) at load,
not just FII/DII.

### C2. Live ATM IV / IV-skew wiring is dead code
**File:** `src/helion_risk_world/data/regime_context_builder.py:195-198` (`set_live_iv`), `src/helion_risk_world/data/model_input_builder.py:201,274`

`set_live_iv()` is never called from any production path — `ModelInputBuilder.build()`/`build_many()` call
`regime_builder.build(ts)` directly with no live option-chain fetch. The only caller in the repo is a unit test.
Contrary to the README's claim ("ATM IV, IV skew ← Upstox live option chain, Live inference only"), predictions
always source IV from `daily_context.parquet`, which only has bhavcopy coverage Jan 2023–Jul 2024 (per C1's
sibling finding, see M-Pred-2 below); outside that window IV features are silently zero everywhere, including
all real "live" runs. This isn't the hypothesized train/serve skew — the feature is simply inert end-to-end.

**Fix direction:** wire a live option-chain fetch into the prediction path, or remove the README claim and treat
IV/skew as historical-only features until it's built.

## HIGH

### H1. RSSM train/serve distribution mismatch (Architecture)
**File:** `src/helion_risk_world/worlds/rssm.py:120-135`, `src/helion_risk_world/model.py:277-281`, `src/helion_risk_world/training/train_world_model.py`

`RSSM.filter()`'s own docstring warns a single-step posterior call is wrong because `h_t` depends on full
history. Yet `HRWWorldModel.forward()` calls `filter()` with a T=1 window every time, always starting from a
zero `initial_state`. Training (`WorldModelTrainer.encode_sequence` / `RSSMLoss`) rolls the recurrence over
T=5+ real steps. The deployed `h_t` is drawn from a distribution the prior/posterior/decode heads were never
fit on — directly undermining the claim that ensemble spread is "calibrated epistemic uncertainty."

### H2. Continuous-futures roll boundary driven by file overlap, not expiry (Data)
**File:** `src/helion_risk_world/data/continuous_futures.py:97-126`, `scripts/fetch_upstox.py:195-221`

The roll point is `next_start = next_df.index.min()` — wherever the next contract's downloaded file happens to
start — not N days before the near contract's expiry. Per-contract files can span their entire listed life, so
the adjustment ratio (`near_close/next_close`) can be computed from non-contemporaneous prices when there's no
true overlap window. `test_build_continuous_two_contracts` uses a 15-day gap and the code proceeds anyway. Any
drift over that gap compounds backward through every earlier contract via `accumulated_ratio`.

### H3. Corporate-action blackout row removal creates an unguarded temporal gap (Data)
**File:** `scripts/assemble_data.py:80-87`, `src/helion_risk_world/labeling/barrier_labeler.py:76-106`, `scripts/label.py:53-98`

Blackout rows are dropped entirely rather than reindexed with placeholders. The barrier labeler uses
**positional** indices (`t+1`, `t+H`) as a proxy for elapsed bars with no check that consecutive rows are
actually adjacent bars. For the ~H decision bars preceding the ~10-day deleted HDFC blackout window, `entry_i =
t+1` silently jumps ~10 days forward, producing entry price / barrier scan / exit_return computed across a
fabricated "1-bar" step that's actually 10 days of market history. Same failure mode applies to any other
dropped-row gap, not just HDFC.

### H4. Roll-gap bars are fabricated, not excluded, and the flag is never consumed (Data)
**File:** `src/helion_risk_world/data/rollover.py:34-55`, `scripts/assemble_data.py:76`

`flag_and_clip().ffill()` fabricates a synthetic flat bar at every roll (close copied forward while open/high/low
are left inconsistent with it). The resulting `roll_gap` boolean survives into the assembled parquet but is
never read by `market_window_builder.py`, `feature_builder.py`, or `barrier_labeler.py` — no rolling window,
return, or barrier scan skips or masks it, so a fabricated flat bar quietly enters realized_vol, ATR, momentum,
and label computations around every single roll.

### H5. Optimization loops have no NaN/Inf guard (Training)
**File:** `src/helion_risk_world/training/trainer.py:134-190`, `train_heads.py:92-113`, `train_world_model.py:95-114`

None of the training loops check `loss.isfinite()` or skip a bad batch before `backward()`/`optim.step()`.
`clip_grad_norm_` doesn't help — a NaN loss (e.g. from a data gap) produces a NaN grad norm, and `optim.step()`
silently corrupts every parameter for the rest of the run with no error and no log signal beyond a NaN history
entry. Realistic given real gaps/holidays in NSE 5-min data.

### H6. Stage-2 pretraining task is near-identity by default (Training)
**File:** `configs/v1.yaml:101` (`pretrain_gap_bars: 1`), `scripts/train.py:522-529` (`_build_pretrain_pairs_from_history`)

With `lookback_bars=96` and `gap_bars=1`, context and future windows overlap 95/96 bars (99%). The
self-supervised latent-prediction objective is trained on two windows differing by one bar, producing
near-identical embeddings — the objective is technically leak-free (gap ≥ 1) but defeats its own purpose of
learning to predict a materially-future latent state.

### H7. `HeadTrainer` (documented Stage 4) is dead code (Training)
**File:** `src/helion_risk_world/training/train_heads.py`, README.md:134,209

README documents `train_heads.py`/`HeadTrainer` (freeze-encoder, fine-tune-heads) as part of the wired Stage-4
pipeline, but no CLI script (`scripts/train.py`, `train_workflow.py`) ever imports it — only `HRWTrainer` runs
end-to-end training. The freeze/fine-tune path is exercised only by unit tests, never by a real training run.

### H8. Backtest's "leakage-checked" run only performs the shallow check (Evaluation)
**File:** `scripts/backtest.py:81`, `src/helion_risk_world/backtesting/leakage_report.py:36-50`

`LeakageReport().run(list(CANDLE_FEATURE_NAMES))` is called with `feature_rows=None, labels=None`. Both the
real per-row point-in-time check (`available_at <= ts`) and the label-future check (`label_realized_at > ts`)
are wrapped in `if ... is not None` guards and never execute with these arguments — only the static "no
portfolio field names in the feature list" check runs. The engine calls this a "leakage-checked backtest," but
a genuine future-peeking bug in the feature-window builder would go completely undetected.

### H9. Epistemic-uncertainty gating is hardcoded off for the plain forecaster (Prediction)
**File:** `src/helion_risk_world/inference.py:213`, `src/helion_risk_world/planner/management_loop.py:58`, `risk/constraints.py:207`, `planner/position_sizer.py:24`

`ForecasterPredictor` hardcodes `epistemic=0.0` for every prediction (no RSSM ensemble to derive it from). Three
downstream safety mechanisms — force-exit on high epistemic, risk-blocking on high epistemic, and
epistemic-scaled position sizing — are all silently inert for any deployment running the plain forecaster
(the default whenever `model_kind != "world_model"`).

### H10. Data-quality fail-safe defaults to off in code (Prediction)
**File:** `scripts/paper_trade.py:98`, `src/helion_risk_world/memory/data_quality_memory.py`

`data_monitor_cfg = paper_cfg.get("data_monitor", False)` — any config lacking an explicit
`paper.data_monitor: true` runs with zero staleness/point-in-time gating. The shipped
`configs/paper_trading.yaml` does set it true, masking the risk today, but it's a silent footgun for any custom
config, and even when enabled the guard only blocks risk-increasing actions for one bar rather than halting the
run.

### H11. `epistemic` is NaN when `n_samples=1` (Architecture)
**File:** `src/helion_risk_world/worlds/market_world.py:183`

`roll.std(dim=0)` with a single ensemble sample is `unbiased=True` 0/0 → NaN. `RolloutEngine.rollout` only
rejects `n_samples < 1`, so `n_samples=1` is a legal call. NaN comparisons are always False, so a risk-shield
gate comparing `epistemic > threshold` would silently fail to trigger instead of flagging risk — compounds with
H9's gating dependency.

## LOW

- **Architecture:** `RSSM.imagine` (`worlds/rssm.py:137-157`) is dead code duplicating `RolloutEngine.rollout` — DRY violation/maintenance risk if the two diverge.
- **Architecture:** `RSSMLoss.l_imag` isn't normalized per (t,k) valid-target count — early sequence positions are implicitly overweighted for longer sequences.
- **Architecture:** `ModelConfig.fusion` type allows `"attention"|"moe"|"uncertainty"` but `FusionEncoder` only implements `"gated"` (raises `NotImplementedError` — fails loud, low risk).
- **Training:** `embargo_bars` isn't cross-validated against `max(target_horizons)` for world-model multi-horizon training specifically (single-horizon path is correctly patched in `scripts/train.py:1729`).
- **Training:** README config-defaults table says `training.lr` default is `1e-3`; code/config actually use `3e-4`. Cosmetic only.
- **Data:** Positional rolling windows (`market_window_builder.py`/`feature_builder.py`) assume uniform bar spacing — not a leakage bug, but holes silently dilute "N-bar" windows into longer wall-clock spans.
- **Data:** `corporate_actions.py:19` comment says "5 trading days" blackout but the code does calendar-day arithmetic — actual window is ~3-4 trading days each side. Low practical impact given the merger date fell on a weekend.
- **Prediction:** Heuristic (non-model) backtest path recomputes full-history rolling features per step — O(N²) over a long replay; model-backed path is unaffected.
- **Prediction:** `ModelPrediction` JSON keys for `return_quantiles`/`regime_probs` round-trip correctly through pydantic v2, but non-Python consumers must parse string-float keys themselves.

## What's solid (verified, not just assumed)

- **Quantile monotonicity** is structurally enforced (`base + cumsum(softplus(...))`), not just encouraged.
- **Pinball loss, soft-coverage loss, heteroscedastic NLL** are textbook-correct.
- **Dreamer v2 KL balancing direction** in `kl_balanced` is correct (0.8/0.2 split trains prior vs posterior as intended) despite confusing variable naming.
- **`FusionEncoder`'s zero-padding** for missing input planes is architecturally safe — no NaN/leakage risk from missing slots.
- **`fit_ood`/`OODHead.fit`** correctly avoid gradient contamination via `@torch.no_grad()`.
- **Split methodology** (`ChronoSplitManifest`, `PurgedKFold` walk-forward path, `WalkForward.split_indices`) is genuinely time-ordered with embargo applied at both boundaries — no naive random split found anywhere in the main training path.
- **Sample-weight plumbing** (uniqueness weights → `sample_weight` column → loss reduction) is fully wired through to gradients, not silently discarded.
- **Encoder freezing correctness** in `HeadTrainer` (when it runs) correctly excludes frozen params from the optimizer's parameter list.
- **Checkpoint/config coupling** is handled well — artifacts persist `ModelConfig` and `input_contract`, and loading raises loudly on mismatch rather than silently producing garbage.
- **Barrier entry/exit mechanics, EWMA vol scaling, and core rolling primitives** (`realized_vol`, `atr`, `bb_position`, `rsi`, etc.) are genuinely causal with no off-by-one leak into the label window.
- **Quantile coverage math, barrier Brier/ECE, class-index alignment** (stop/target/timeout) are all correct end-to-end from `BarrierHead` through `calibrate.py`.
- **Portfolio consequence model** correctly carries forward existing position state (capital, margin, drawdown) rather than treating each step independently.
- **Transaction costs, spread, slippage, and risk-shield vetoes** are genuinely applied to simulated P&L in the backtest engine, not computed and discarded.
- **OOD score, risk vetoes, and point-in-time timestamp handling** are genuinely wired end-to-end in the paper-trading/backtest planner path (separate from the C2/H9 gaps noted above, which are specific dead-wiring instances).

## Priority recommendation

If only a handful of items can be fixed before the next training/backtest cycle, address in this order:

1. **C1** (daily_context same-day leak) — invalidates backtest performance claims until fixed; highest ROI.
2. **H8** (backtest leakage check is toothless) — currently the only thing that would have caught #1 and #2/#4, and it's disabled.
3. **H3 + H4** (blackout/roll-gap positional-index corruption) — corrupts labels around every HDFC-style event and every futures roll, not a rare edge case.
4. **H1** (RSSM train/serve mismatch) — only matters once/if the world-model path is promoted to production; currently a research path.
5. **C2 + H9 + H10** (dead IV wiring, disabled epistemic gating, disabled data-quality monitor by default) — three independent "safety feature that looks wired but isn't" bugs in the live/paper path; worth a single pass to audit all config-default-driven safety toggles.
